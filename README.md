# DevPay

---

Асинхронный бэкенд-сервис для автоматизации и управления платными подписками в Telegram-ботах с фокусом на отказоустойчивость, надежность доставки данных и безопасность.

---

## 🛠 Технологический стек

* **Язык разработки:** Python 3.11+ (asyncio)
* **Фреймворк:** FastAPI
* **Базы данных и кэширование:** PostgreSQL, SQLAlchemy 2.0 (Asyncio / asyncpg), Redis (Connection Pools)
* **Брокер сообщений:** RabbitMQ (aio-pika)
* **Управление задачами и миграциями:** APScheduler, Alembic
* **Управление зависимостями:** Poetry
* **Тестирование:** Pytest (AsyncMock)
* **Инфраструктура:** Docker, Docker Compose

---

## 🏗 Архитектурные решения и особенности проекта

Проект спроектирован с учетом лучших практик построения распределенных отказоустойчивых систем и решает ключевые боли бэкенд-разработки:

1. **Гарантированная доставка событий (Transactional Outbox):**
   Платежные вебхуки и бизнес-события записываются в PostgreSQL в рамках единой транзакции с обновлением статусов подписок. Изолированный воркер считывает таблицу Outbox и отправляет сообщения в RabbitMQ, что полностью исключает потерю данных при падении брокера.

2. **Двухуровневая защита от повторных списаний (Идемпотентность):**
   Для защиты от дублирования сетевых запросов и вебхуков платежных шлюзов (симулировалась работа с ЮMoney) реализован механизм фильтрации:
   * Быстрый In-Memory фильтр в Redis с настраиваемым TTL.
   * Строгая проверка уникальности транзакций (Unique Constraints) на уровне базы данных PostgreSQL.

3. **Защита от состояний гонки (Race Conditions):**
   Смена статусов в стейт-машине подписок (создана -> активна -> просрочена -> отменена) обрабатывается фоновым воркером асинхронно. Конкурентный доступ нескольких экземпляров воркеров к одним и тем же записям заблокирован на уровне СУБД с помощью механизма `SELECT FOR UPDATE SKIP LOCKED`.

4. **Криптографическая безопасность (SecOps):**
   Чувствительные пользовательские данные (платежные реквизиты, токены мерчантов) шифруются перед сохранением в PostgreSQL с использованием симметричного шифрования Fernet / MultiFernet, что позволяет проводить бесшовную ротацию ключей шифрования без остановки сервиса.

## 📁 Структура проекта

```text
├── migrations/             # Миграции базы данных (Alembic)
├── src/                    # Исходный код приложения
│   ├── api/                # Эндпоинты FastAPI (REST API v1)
│   ├── cron/               # Планировщик фоновых задач (APScheduler)
│   ├── services/           # Бизнес-логика, шифрование, кэш, брокенг
│   ├── config.py           # Конфигурация приложения (Pydantic Settings)
│   ├── main.py             # Точка входа для запуска API
│   ├── models.py           # Описание моделей SQLAlchemy (PostgreSQL)
│   └── exceptions.py       # Кастомные исключения приложения
├── worker.py               # Background-воркер для обработки очередей RabbitMQ
├── docker-compose.yml      # Инфраструктурное окружение приложения
├── pyproject.toml          # Зависимости проекта (Poetry)
└── alembic.ini             # Конфигурация Alembic
```

## 🚀 Запуск проекта
1. **Клонирование репозитория и настройка окружения**

```text
git clone https://github.com/fl1k3r-dev/DevPay.git
cd DevPay
```
Создайте файл .env в корневой директории по шаблону (проект использует автоматическую валидацию переменных окружения через Pydantic). Примечание: .env добавлен в .gitignore и не попадает в публичный доступ.

2. **Запуск инфраструктуры через Docker Compose**
Команда развернет все необходимые сервисы (API, PostgreSQL, RabbitMQ, Redis, Worker):

```text
docker-compose up -d --build
```

3. **Применение миграций базы данных**
```text
poetry run alembic upgrade head
```

## 🧪 Тестирование
Бизнес-логика и сценарии работы стейт-машины полностью покрыты тестами. В проекте реализовано два подхода:

Offline-тестирование: Изолированные юнит-тесты бизнес-логики с использованием AsyncMock (без поднятия реальной инфраструктуры).

Online-тестирование: Полноценные интеграционные тесты стейт-машины и очередей событий в реальном Docker-окружении.

Запуск тестов:

```text
poetry run pytest
```
